# Node backend с нуля — эволюционный роадмап

Цель: снять фреймворк и пройти путь, который ecosystem Node проделал сам —
от голого `node:http` до DI-контейнера и request pipeline. На каждом слое
ты сначала **чувствуешь боль**, которую этот слой решает, и только потом
добавляешь абстракцию. Так становится видно, что именно Nest делает за тебя
и какой ценой.

---

## Правила игры

- **Домен фиксирован, меняется только обвязка.** Фичи определяешь один раз
  (см. ниже) и на каждой фазе переписываешь *только инфраструктуру* вокруг них.
  Одна и та же задача, решённая на 7 уровнях абстракции — в этом вся суть.
- **Не прыгай вперёд.** Не тащи Express, пока руками не написал роутинг.
  Не тащи DI-контейнер, пока не собрал граф руками. Боль — это учебный материал.
- **Git-тег на каждую фазу** (`phase-0`, `phase-1`, …). Потом `git diff` между
  фазами — половина инсайтов именно там.
- **3 строки рефлексии в конце каждой фазы:** что этот слой убрал из кода
  предыдущего и что взамен спрятал. Пиши прямо в README.
- Оценки времени ниже — грубые, для senior-темпа в режиме pet-project.

## Фиксированный домен: мини-Booking API

Маленький, но с настоящей логикой, чтобы слои имели смысл. Не расширяй его —
это соблазн, который убьёт фокус.

Ресурсы и операции:

- `POST /bookings` — создать бронь на слот (`slotId`, `userId`)
- `GET /bookings` — список
- `GET /bookings/:id` — одна бронь
- `POST /bookings/:id/cancel` — отменить

Один **инвариант**: слот нельзя забронировать дважды (пока бронь активна).
Один **доменный error**: попытка двойного бронирования → конфликт (409).
Этого достаточно, чтобы прочувствовать валидацию, доменные ошибки,
транзакции и конкурентность — но мало, чтобы утонуть.

## Фаза 0 — Голый `node:http`

**Цель.** Понять, что `req`/`res` — это стримы, и что весь Express/Fastify —
надстройка над этим. Никакой магии, всё руками.

**Что делаешь.**

Ниже — каждый пункт с расшифровкой: *что* сделать и *зачем именно*, чтобы не
было простора для «я понял иначе».

- **Поднять сервер на `node:http`.** Только встроенный модуль ноды, ни одной
  npm-зависимости в рантайме. `@types/node` и сам TypeScript-компилятор не в
  счёт — это dev-инструменты, они не попадают в работающий процесс. Смысл: увидеть
  голый HTTP-сервер до того, как его спрячет фреймворк.

- **Маршрутизация руками.** Написать диспетчер, который по паре
  «`req.method` + путь из `req.url`» находит нужный хендлер. Отдельно —
  разбор *path-параметра*: в маршруте `/bookings/:id` кусок `:id` это не буквальный
  текст, а переменная. Для запроса `GET /bookings/abc-123` диспетчер должен понять,
  что это маршрут `/bookings/:id`, и достать `id = "abc-123"`, отдав его хендлеру.

- **Парсинг тела руками.** Тело POST-запроса приходит не целиком, а кусками
  (`data`-чанки стрима) — надо их накопить в строку, дождаться конца (`end`) и
  прогнать через `JSON.parse`. Обязательно обработать случай, когда клиент прислал
  **не-JSON** (например, `curl -d 'мусор'`): `JSON.parse` кинет исключение, и это
  исключение нужно поймать и превратить в аккуратный ответ `400`, а **не** дать
  ему упасть в общий обработчик и стать `500`. `400` = «ты, клиент, прислал
  ерунду»; `500` = «это я, сервер, сломался». Битое тело — вина клиента, поэтому `400`.

- **Ответ руками.** Самому выставить код статуса, заголовки (как минимум
  `Content-Type: application/json`) и закрыть ответ через
  `res.end(JSON.stringify(...))`. Никаких `res.json()` — их ещё нет, это как раз то,
  что позже добавит фреймворк.

- **Правильные коды ответов.** Явно и осознанно:
  - `404` — запрошен маршрут, которого нет (`GET /чего-то-нет`), либо бронь по `id`
    не найдена.
  - `400` — тело запроса кривое (не парсится) или в нём не хватает обязательных
    полей.
  - `409` (Conflict) — попытка нарушить доменное правило, а именно **двойная бронь**
    (см. следующий пункт). Это *отдельный* код, не путать с `400`: тут запрос
    синтаксически корректен, но конфликтует с текущим состоянием данных.

- **Инвариант «двойная бронь» — читать внимательно.** Правило такое: *один слот
  не может быть занят двумя активными бронями одновременно*. Ключевое слово — **слот**,
  а не пара «слот+пользователь». То есть проверка при создании брони: «существует ли
  уже **активная** бронь на этот `slotId`?» — и если да, отдать `409`. Пользователь
  (`userId`) в этой проверке роли не играет: если слот №1 уже забронирован, его не может
  занять *никто* другой, пока та бронь активна. Слово «активная» тоже важно: если бронь
  на слот отменили (`cancel`), слот снова свободен и его можно бронировать заново.
  Частая ошибка — проверять по `slotId` *и* `userId`: тогда чужой человек сможет занять
  уже занятый слот, и инвариант сломан.

- **Хранилище — в памяти.** Обычный `Map` (ключ — `id` брони). Никакой базы, всё
  живёт, пока процесс запущен. На этой фазе так и задумано.

**Что должно получиться.** Рабочий CRUD-API на чистой ноде, ноль внешних зависимостей.

**Промежуточные этапы.**
1. Сервер отвечает «hello» на любой запрос.
2. Диспетчер маршрутов (метод + путь → хендлер) с path-параметром.
3. Парсинг тела + отдача JSON с корректным статусом (включая `400` на битый JSON).
4. Все 4 эндпоинта + инвариант двойной брони (по слоту, с учётом отмены) на `409`.

**Критерии завершения.**
- [ ] Все 4 эндпоинта работают через `curl` и возвращают ожидаемые коды:
      `201` на создание, `200` на чтение/список, `404` на несуществующее,
      `400` на кривое тело, `409` на двойную бронь.
- [ ] Двойная бронь одного слота (разными пользователями) отклоняется с `409`;
      после `cancel` слот снова можно забронировать.
- [ ] Битый JSON в теле даёт `400`, а не `500`.
- [ ] Ноль рантайм-зависимостей в `package.json` (только `devDependencies`).
- [ ] Ты можешь на словах пройти путь «байты пришли → распарсили → хендлер →
      ответ ушёл» без пробелов.

_Оценка: 0.5–1 день._

## Фаза 1 — Собери свой мини-Express

**Цель.** Понять, *зачем существует Express*, построив его ядро самому. После
этой фазы ты никогда не будешь смотреть на middleware как на чёрный ящик.

**Что делаешь.**
- `Router`: регистрация хендлеров по методу/пути, матчинг path-параметров.
- Middleware-конвейер: цепочка `(req, res, next)`, вызов по порядку.
- Middleware парсинга тела (выносишь логику из Фазы 0 в переиспользуемый слой).
- Error-handling middleware (сигнатура с `err` первым аргументом — как в Express).
- Хелперы на ответе: `res.status().json()`.
- Переписать приложение Фазы 0 поверх этого «фреймворка».

**Что должно получиться.** Модуль-микрофреймворк + то же приложение на нём,
но код стал декларативным.

**Промежуточные этапы.**
1. Router с path-параметрами.
2. Конвейер middleware работает и порядок соблюдается.
3. Error-middleware ловит брошенное из хендлера.
4. Приложение переехало на свой фреймворк, поведение не изменилось.

**Критерии завершения.**
- [ ] Поведение идентично Фазе 0, но в коде приложения нет ручного разбора
      url/тела.
- [ ] Порядок middleware управляет поведением (проверь на logging-middleware).
- [ ] Можешь сформулировать, что такое «ядро Express» в одном абзаце.

_Оценка: 1 день._

## Фаза 3 — Слои и composition root (ручной DI)

**Цель.** Разнести ответственность и собрать граф зависимостей руками. Понять,
что DI-контейнер **не обязателен**, а ручная сборка часто *чище*, потому что
граф виден глазами.

**Что делаешь.**
- Разбить на слои: `controller` (http) → `service`/use-case (логика) →
  `repository` (интерфейс) + in-memory реализация.
- Инвариант двойной брони живёт в домене/сервисе, а не в контроллере.
- **Dependency inversion:** сервис зависит от *интерфейса* репозитория, не от
  реализации.
- Вся проводка — в одной точке (`composition root` / `bootstrap`). Контейнера нет.

**Что должно получиться.** Слоёное приложение, весь граф зависимостей собран
явно в одном месте.

**Промежуточные этапы.**
1. Выделен интерфейс репозитория.
2. Выделен сервис, инвариант переехал в домен/сервис.
3. Проводка в composition root.
4. Подменил in-memory репозиторий на вторую реализацию (например, «репозиторий с
   логированием»), чтобы доказать, что шов работает.

**Критерии завершения.**
- [ ] Реализацию репозитория можно поменять, отредактировав *одну строку* в root.
- [ ] Бизнес-правило ничего не знает ни про http, ни про хранилище.
- [ ] Контроллер тонкий: разобрал вход → вызвал use-case → отдал ответ.

_Оценка: 1 день._

## Фаза 4 — Свой DI-контейнер (reflect-metadata + декораторы)

**Цель.** Распотрошить главную «магию» Nest: декораторы просто пишут метадату,
а бутстрап её потом читает и строит граф. Это фаза «а-а-а, вот как оно работает».

**Что делаешь.**
- Включить `reflect-metadata`, в `tsconfig`: `experimentalDecorators` +
  `emitDecoratorMetadata`.
- `@Injectable()` — маркер класса.
- Контейнер, который через `Reflect.getMetadata('design:paramtypes', Target)`
  достаёт типы параметров конструктора и **рекурсивно** резолвит зависимости.
- Кэш синглтонов (один инстанс на токен).
- Регистрация по токену (для интерфейсов, у которых нет рантайм-типа —
  через строковый/Symbol-токен + `@Inject`).
- _(stretch)_ `@Controller`/`@Get`, которые пишут метадату маршрутов, а бутстрап
  их читает и регистрирует роуты автоматически.

**Что должно получиться.** Работающий DI-контейнер + приложение, проводка
которого идёт через него вместо ручного root.

**Промежуточные этапы.**
1. `@Injectable` + резолв одного сервиса.
2. Рекурсивный резолв всего графа.
3. Скоуп-синглтон (кэш инстансов).
4. _(stretch)_ Детект циклических зависимостей с внятной ошибкой.
5. _(stretch)_ Метадата-роутинг: `@Get('/bookings/:id')` регистрирует хендлер.

**Критерии завершения.**
- [ ] Регистрируешь класс — контейнер сам поднимает весь его граф.
- [ ] Можешь по шагам объяснить, что именно Nest делает в момент bootstrap
      (сканирование → метадата → построение графа провайдеров → роуты).
- [ ] Понимаешь, *почему* для интерфейсов нужны токены, а для классов — нет.

_Оценка: 1.5–2 дня._

## Фаза 5 — Request pipeline (validation, guards, interceptors, filters)

**Цель.** Пересобрать сквозной конвейер, который Nest даёт из коробки, и увидеть,
что каждый его «магический» элемент — это просто функция в нужной точке.

**Что делаешь.**
- **Валидация** на границе через Zod (аналог `ValidationPipe`): невалидный вход →
  400 с внятным телом.
- **Guard-аналог**: проверка до хендлера (например, простая проверка `userId` /
  «авторизации») → 401/403.
- **Interceptor-аналог**: обёртка вокруг хендлера (логирование + тайминг,
  трансформация ответа).
- **Exception filter**: централизованный маппинг доменных ошибок → HTTP
  (двойная бронь → 409, не найдено → 404) вместо `if`-ов по коду.
- Собрать порядок: `guard → validate → handler → filter`, обёрнутый interceptor’ом.

**Что должно получиться.** Композируемый pipeline вокруг хендлеров; в контроллерах
больше нет ad-hoc проверок и ручного маппинга ошибок.

**Промежуточные этапы.**
1. Слой валидации (Zod-схемы на вход).
2. Guard.
3. Interceptor (логирование/тайминг).
4. Exception filter + доменные классы ошибок.
5. Сборка конвейера в правильном порядке.

**Критерии завершения.**
- [ ] Невалидный вход, «неавторизован» и нарушение доменного правила дают
      корректные консистентные ответы — через pipeline, а не через разрозненные `if`.
- [ ] Контроллер не знает, как ошибка домена превращается в HTTP-код.
- [ ] Можешь сопоставить каждый элемент своего конвейера с pipe/guard/interceptor/
      filter из Nest.

_Оценка: 1.5 дня._

## Фаза 6 — Реальная персистентность и жизненный цикл

**Цель.** Честная БД и продакшн-lifecycle без магии ORM.

**Что делаешь.**
- Заменить in-memory репозиторий на Postgres через `pg` (или Kysely/Drizzle) —
  явный SQL, явные транзакции.
- Инвариант двойной брони закрепить **и на уровне БД**: `UNIQUE`-constraint на
  активный слот + корректная обработка конфликта (ловишь ошибку уникальности →
  доменный 409). Это заодно решает конкурентность, которую in-memory прятал.
- Пул соединений + конфиг через env.
- **Graceful shutdown** на `SIGTERM`/`SIGINT`: перестать принимать новые запросы,
  дренировать текущие, закрыть пул.
- Структурированное логирование, healthcheck-эндпоинт.

**Что должно получиться.** Приложение, работающее с реальным Postgres и корректно
завершающееся.

**Промежуточные этапы.**
1. Пул `pg` + конфиг.
2. Репозиторий на SQL.
3. Транзакция вокруг создания брони.
4. `UNIQUE`-constraint + обработка конфликта.
5. Graceful shutdown.
6. Healthcheck.

**Критерии завершения.**
- [ ] Kill-сигнал завершает процесс чисто, без оборванных соединений.
- [ ] Две одновременные попытки забронировать один слот → одна проходит, вторая
      отклонена БД-констрейнтом (проверь гонку, а не только последовательные запросы).
- [ ] Транзакции и границы явные, ты контролируешь их руками.

_Оценка: 1.5–2 дня._

## Фаза 7 (сведение) — Сравнение с Nest

**Цель.** Кристаллизовать всё, что понял: построить карту «каждая абстракция Nest →
сырой механизм под ней».

**Что делаешь.**
- Быстро (ты это умеешь) переписать то же приложение на Nest.
- Написать сравнение-карту:
  - module ↔ composition root
  - provider ↔ регистрация в контейнере
  - pipe ↔ слой валидации
  - guard ↔ guard
  - interceptor ↔ interceptor
  - exception filter ↔ exception filter
  - lifecycle hooks (`onModuleInit`, shutdown hooks) ↔ твой graceful shutdown
- Отдельно отметить: где ручной вариант вышел чище, а где Nest реально отбивает
  свою цену.

**Что должно получиться.** Короткий раздел README / сравнительный документ.

**Критерии завершения.**
- [ ] Для каждой абстракции Nest ты можешь назвать сырой механизм под ней.
- [ ] Есть осознанное мнение, где фреймворк оправдан, а где — оверхед.

_Оценка: 0.5–1 день._

## Кроличьи норы (опционально, если зацепит)

Не обязательны для основной цели, но каждая — отдельный глубокий инсайт:

- **Стримы и backpressure**: почему наивный парсинг тела ломается на больших
  payload’ах; `pipeline`, `highWaterMark`.
- **Keep-alive и пул соединений на уровне HTTP**: как ведёт себя `http.Agent`.
- **Кластеризация**: `cluster` / несколько воркеров + балансировка, и почему это
  не то же самое, что `worker_threads`.
- **HTTP/2**, graceful reload без даунтайма.
- **Свой минимальный тест-раннер** поверх `node:test` — понять, что даёт Jest.
